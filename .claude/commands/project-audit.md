# Project Audit

You are the **lead** of Testopus's senior engineering council. Convene the team to audit the
project across three fixed deliverables, let the experts exchange and challenge findings, and
converge on documents the team can act on. Stay neutral as facilitator; the `tech-lead` agent owns
the final synthesis.

## Mission

Audit the project (scope below) and produce **three** documents in `docs/audit/`:

1. **Antipatterns** (`docs/audit/antipatterns.md`) — find design/code/process antipatterns and
   convention violations. For each: location (`file:line`), the principle/convention it breaks, its
   impact, and a concrete **action item** with an owner role. If an area is clean, say so.
2. **Roadmap** (`docs/audit/roadmap.md`) — improvements and new features, phased **Now / Next /
   Later**. For each item state **what it affects** (files/areas) and **what value it delivers**.
3. **Best practices** (`docs/audit/best-practices.md`) — industry/framework best practices the
   project should adopt. For each: the practice, the current gap (`file:line`), how to adopt it, and
   a reference to official docs where relevant.

## Scope

$ARGUMENTS

If the scope above is empty, audit the **whole repository** (`core`, `fixtures`, `utils`, `config`,
`tests`, `ci`, `docker`, `pyproject.toml`, the `.claude/` toolkit).

## Roster (each is a `.claude/agents/*.md` agent type)

`qa-automation-engineer`, `backend-engineer`, `frontend-engineer`, `devops-engineer`,
`ai-engineer`, `ml-engineer`, `data-engineer`, `architect` — and `tech-lead` as synthesizer.

## Mode (auto-detect)

Experimental agent-teams flag: !`printenv CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS 2>/dev/null || echo "(unset)"`

- If the value above is **`1`** → run in **agent-teams** mode: assemble a team and spawn each role as
  a teammate **using its agent type**, so teammates can message each other directly and share a task
  list. Note for the user that this costs ~7× tokens and is experimental (no `/resume` of teammates,
  shutdown can lag). Keep the team to these 9 roles; don't nest teams.
- Otherwise → run in **subagent fan-out** mode: you (the lead) spawn each role with the Agent tool,
  round by round, and relay peers' findings between rounds. Cheaper and deterministic.
- **Fallback:** if a role's agent type doesn't resolve as a subagent, spawn `general-purpose` instead
  and have it read and adopt that role's `.claude/agents/<role>.md` charter before auditing.

Either way the round structure below is identical.

## Rounds

**Round 0 — Frame.** Restate the audit scope in one paragraph. Read just-enough repo context
(`CLAUDE.md`, `pyproject.toml`, `core/`, `fixtures/`, `utils/`, `config/`, `tests/`, `ci/`,
`docker/`, `.github/`). State the **success criteria** for a useful audit and assign each role its
lens. Keep scope tight.

**Round 1 — Independent audit.** Have all 8 role agents respond **in parallel**, each from its own
lens with no knowledge of the others yet. Require each to return three lists, every entry citing
`file:line`: **Antipatterns** (+ violated principle), **Roadmap items** (+ what it affects + value),
**Best practices** (+ current gap + how to adopt).

**Round 2 — Cross-critique & exchange.** Give every agent a digest of the others' findings. Each one
engages by name: confirm or refute findings (kill false positives), de-duplicate overlaps, expose
cross-cutting issues and hidden coupling, and rank severity/priority. Push for real disagreement over
polite consensus — especially qa-automation-engineer vs. ai-engineer/ml-engineer on test
determinism, and architect on keeping AI/ML/data layers optional.

**Round 3 — Synthesis.** Invoke `tech-lead` with the full discussion. It consolidates everything into
the three deliverables: de-duplicated, severity- and phase-ordered, every item attributed to an owner
role, with recorded dissent. Cut speculative scope; keep the minimum that delivers value.

## Output

Write the three documents to `docs/audit/` (`antipatterns.md`, `roadmap.md`, `best-practices.md`),
each opening with a blockquote metadata header (Date · Generated by `/project-audit` · Mode ·
Participants), matching the house style of `.claude/council/*.md`. Then summarize the headline
findings to the user and point to the files. Do **not** implement fixes from this command — surface
the action items and let the user (or a follow-up) execute them, with the mandatory `code-reviewer`
gate before any push.

## Guardrails

KISS, DRY, SOLID; minimum non-speculative proposals; cite `file:line` for every finding. Docs-first:
when a finding touches Selenium/Pytest/Allure/Playwright/Appium/Hatch or the Anthropic SDK, verify
against official docs before asserting an API fact. Surface tradeoffs and dissent instead of hiding
them. Don't duplicate the enabled `code-review` / `playwright` / `claude-md-management` plugins —
reference them.

## Example invocations

- `/project-audit` — audit the whole repository.
- `/project-audit CI pipeline and Allure publishing` — focus on `.github/workflows/`, `ci/`, `docker/`.
- `/project-audit the POM layer` — focus on `core/pom/web/` conventions and resilience.
